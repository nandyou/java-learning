Spring Controller Mappings

Spring provides dedicated annotations for each HTTP verb and powerful tools to extract data from URLs, query strings, request bodies, and headers.

import org.springframework.web.bind.annotation.*;
import org.springframework.http.ResponseEntity;
import org.springframework.http.HttpStatus;
 
@RestController
@RequestMapping("/api/users")
public class UserController {
 
// ① GET — retrieve all resources
@GetMapping
public List<User> getAllUsers() {
return userService.findAll();
}
 
// ② GET — retrieve one by path variable
@GetMapping ("/{id}")
public User getUserById( @PathVariable Long id) {
return userService.findById(id);
}
 
// ③ GET — filter with query parameters
@GetMapping ("/search")
public List<User> search(
@RequestParam (value = "name", required = false) String name,
@RequestParam (defaultValue = "0") int page) {
return userService.search(name, page);
}
 
// ④ POST — create a resource
@PostMapping
public ResponseEntity <User> createUser( @RequestBody User user) {
User saved = userService.save(user);
return ResponseEntity . status (HttpStatus.CREATED).body(saved);
}
 
// ⑤ PUT — full update
@PutMapping ("/{id}")
public User updateUser( @PathVariable Long id, @RequestBody User user) {
return userService.update(id, user);
}
 
// ⑥ PATCH — partial update
@PatchMapping ("/{id}/email")
public User patchEmail( @PathVariable Long id,
@RequestHeader ("X-New-Email") String email) {
return userService.updateEmail(id, email);
}
 
// ⑦ DELETE — remove a resource
@DeleteMapping ("/{id}")
public ResponseEntity <Void> deleteUser( @PathVariable Long id) {
userService.delete(id);
return ResponseEntity . noContent ().build();
}
}

@GetMapping
Handles HTTP GET requests. Safe and idempotent — calling it multiple times has no side effects.
Shorthand for @RequestMapping(method = RequestMethod.GET).
Use for fetching data. Never use GET to modify server state.
Path Variable Template ( /{id} )
Curly braces define a named placeholder in the URL path.
Example: a request to /api/users/42 binds 42 to the id parameter.
Multiple variables: "/orders/{orderId}/items/{itemId}"
@PathVariable
Extracts the value from the URL path and binds it to the method parameter.
The parameter name must match the template variable name, or use @PathVariable("id").
Spring auto-converts to the declared type: Long, int, UUID, etc.
@RequestParam
Extracts a query parameter from the URL: /search?name=Alice&page=2.
required = false makes it optional (defaults to null).
defaultValue provides a fallback when the param is absent.
For many params, bind them all to a DTO object instead.
@PostMapping
Handles HTTP POST requests. Used to create a new resource.
Not idempotent — calling it twice creates two resources.
Convention: return 201 Created with the new resource in the body.
ResponseEntity<T>
Gives full control over the HTTP response: status code, headers, and body.
More expressive than returning just the object.
Common builders: .ok(body), .status(201).body(obj), .notFound().build(), .noContent().build()
@RequestBody
Deserialises the HTTP request body (JSON by default) into a Java object.
Jackson library performs the JSON → Java conversion automatically.
Add @Valid before it to trigger Bean Validation: @Valid @RequestBody User user
ResponseEntity.status(HttpStatus)
Sets the HTTP status code explicitly. HttpStatus.CREATED = 201.
Chain .body(object) to include a response payload.
Other common statuses: OK(200), NOT_FOUND(404), BAD_REQUEST(400), NO_CONTENT(204)
@PutMapping
Handles HTTP PUT — replaces the entire resource with the provided data.
Idempotent: calling it multiple times with the same data gives the same result.
If a field is missing in the body, it should be set to null/default (full replacement).
@PatchMapping
Handles HTTP PATCH — updates only the specified fields of a resource.
Not idempotent by definition (though implementations can be).
Preferred over PUT when you want to update a single field without sending the full object.
@RequestHeader
Extracts a named HTTP request header and binds it to a method parameter.
Throws MissingRequestHeaderException if the header is absent (add required=false to make it optional).
Common headers: Authorization, Content-Type, Accept, custom headers like X-Request-ID
@DeleteMapping
Handles HTTP DELETE — removes the identified resource.
Idempotent: deleting an already-deleted resource should return the same result (204 or 404).
Convention: return 204 No Content with an empty body on success.
ResponseEntity.noContent()
Builds a 204 No Content response — correct for successful DELETE operations.
.build() finalises it without a body (body is not needed for 204).
Equivalent to: new ResponseEntity<>(HttpStatus.NO_CONTENT)
